# DCC 1.1.0 additive-contracts design

## Purpose

Version 1.1.0 makes DCC usable by two audiences it currently underserves:
non-programmer survey staff, and general-purpose AI systems that drive the
package through its public API. It does so without breaking any 1.0.x caller.
Every capability in this release is *additive*: new exported functions, new
declarative rule and configuration forms, new machine-readable contracts, and
new documentation. No existing finding column, audit column, action name, or
call signature changes meaning.

This release implements the recommended, in-scope items from Phases 3, 4, and 5
of `ENGINEERING_PLAN.md`. It explicitly does not implement the high-risk
boundary functions (general statistical `impute`, arbitrary deductive
correction, psychometric modelling), which remain Planned per section 2.3 of
the engineering plan.

## Version and compatibility strategy

- 1.1.0 is a minor release: additive only. Any script written against 1.0.1
  continues to run unchanged.
- The three workstreams (functionality alignment, survey usability, AI
  usability) are independent and may land in any order across the 1.1.0
  development line. The version is bumped to 1.1.0 once at least the AI-contract
  and one user-facing workstream are complete; the remaining workstream may
  follow in 1.1.x if needed.
- New rule types (`skip_logic`) and configuration forms (Excel cleaning plan)
  are opt-in. A rule file or workflow that does not use them behaves exactly as
  in 1.0.1.
- Capability, schema, and configuration documents are themselves versioned so
  that AI callers can detect the contract level they are working against.

## Public contract by workstream

### Phase 3: functionality alignment

`dcc_apply_codebook(x, codebook, dry_run = TRUE)` applies a declarative
codebook to a `dcc_data` object: variable renames, value recodes, variable and
value labels, missing-value declarations, type requirements, and ID / group /
weight / item role assignments. With `dry_run = TRUE` (default) it returns a
preview describing every change it would make and changes nothing; with
`dry_run = FALSE` it returns a new `dcc_data` version with a `codebook`
provenance record. Raw input is never overwritten. Unknown variables,
impossible type coercions, and recode targets that collide with existing values
are reported as structured errors, never applied silently.

A new declarative rule type expresses common skip logic without embedded R:

```yaml
type: skip_logic
when:
  variable: q1
  equals: 2
then_not_required: [q2, q3, q4]
```

`dcc_detect()` evaluates a `skip_logic` rule into a structural-missingness map:
for each record where the condition holds, the listed variables are marked
*not administered* rather than *omitted*. The missing-items detector consumes
this map so that a legitimately skipped item is not counted as missingness. The
map is exposed through a public accessor and recorded in provenance.

`dcc_to_irtc()` is delivered only if its output contract (`resp`, `pid`,
`group`, `pweights`, `Q`, and anchor information) is stable and end-to-end
testable in this release; otherwise it is documented as Planned and removed
from Stable claims. The Phase 3 plan makes this decision explicitly with a
test-backed contract or a documented deferral.

### Phase 4: survey-staff usability

`dcc_run(data, config, output_dir, mode = "preview")` is the single entry point
survey staff use. It orchestrates the existing engine: read -> validate ->
detect -> (execute) -> report -> manifest, writing a fixed output layout:

```text
<output_dir>/
|-- cleaned-data.csv
|-- findings.xlsx
|-- audit-log.csv
|-- management-report.html
|-- audit-report.html
|-- manifest.yaml
`-- run-summary.txt
```

Supported `mode` values: `preview` (detect and report only; no data change and
no `cleaned-data.csv` overwrite of raw input), `execute` (apply the configured
actions), `verify` (reconcile an existing run), and `rerun` (reproduce from a
manifest). Preview is the default so the safe path requires no extra care. The
raw input file is never overwritten in any mode.

`dcc_read_config(path)` converts an Excel cleaning-plan workbook into the
internal rule set and action map, so staff specify the data file, record ID,
item columns, allowed values, ranges, skip conditions, attention checks,
response-time limits, missingness thresholds, and dispositions in a workbook
rather than writing YAML. A documented template workbook ships with the
package.

`dcc_validate_rules(rules)`, `dcc_validate_data(data, rules)`,
`dcc_validate_config(config)`, and `dcc_doctor(data, config)` each return a
structured validation report: a table of issues with a stable machine code, a
severity, the affected field, the affected rows, and a suggested fix, plus a
human-readable rendering. Validation never changes data and always precedes
execution in `dcc_run()`.

### Phase 5: AI usability

`dcc_capabilities()` returns a versioned, machine-readable description of every
Stable, Experimental, and Planned feature, every rule and action type, and
every input format, matching the capability matrix above. It is pure and
deterministic.

Formal JSON Schemas are published under `inst/schemas/` for the YAML rule file,
the action map, the Excel cleaning plan (as a normalized JSON form), answer
keys, form-item maps, manifests, findings, and audit logs. `dcc_schema(name)`
returns the schema path or parsed content. The schemas are versioned and
regression-tested against real objects the package produces.

Public accessors replace every dependence on hidden attributes:
`dcc_unhandled(result)`, `dcc_item_map(mapped)`, `dcc_mapping_findings(mapped)`,
`dcc_run_files(run)`, and `dcc_validation_errors(report)`. No documented AI
workflow reads an undocumented attribute.

`AI_USAGE.md` documents the minimal safe flow, the approved public functions,
complete rule and action examples, unsupported operations, the
validate-before-execute requirement, success checks, error handling, raw-data
safety, and expected outputs. The documented order is:

```text
capabilities -> validate configuration -> preview findings ->
validate actions -> execute -> reconcile -> export -> verify manifest
```

## Data flow

For survey staff: `dcc_run()` reads the Excel or YAML config, runs
`dcc_validate_*`, calls `dcc_read()` / `dcc_detect()` / (in execute mode)
`dcc_execute()`, then `dcc_report()` and `dcc_manifest()`, and writes the fixed
layout. For AI callers: `dcc_capabilities()` and `dcc_schema()` describe the
contract; the caller validates its configuration, previews findings, validates
actions, executes, reconciles, exports, and verifies the manifest, using only
public accessors. Both paths reuse the same 1.0.1 execution engine, so audit
correctness is inherited, not re-implemented.

## Testing and gates

Each plan is written test-first with a genuine red step before production code.
Coverage additions include: codebook dry-run and apply parity; skip-logic
structural missingness feeding the missing-items detector; `dcc_run()` output
layout and mode semantics with raw-input immutability; Excel-config to
rule/action conversion round-trips; validator message structure and severity;
`dcc_capabilities()` shape and status accuracy; JSON Schema validation of real
findings, audit logs, rules, and manifests; and accessor equivalence with the
attributes they replace.

Release gates for 1.1.0 (in addition to the standing `R CMD check` 0/0/0 and
testthat 0-failure/0-warning/0-skip gates):

- Every new Stable capability has an end-to-end test.
- `dcc_capabilities()` output matches the documented matrix, checked by a test
  that fails when the two diverge.
- Every published JSON Schema validates an object the package actually
  produces, and rejects a deliberately malformed one.
- No documented AI or survey workflow reads a hidden attribute.
- `dcc_run()` never overwrites raw input in any mode, checked by a test.
- README, NEWS, CHANGELOG, and the capability matrix agree with the
  implementation; no Stable claim is unimplemented.
