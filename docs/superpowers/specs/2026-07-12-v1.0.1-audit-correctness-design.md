# DCC 1.0.1 audit-correctness design

## Purpose

Version 1.0.1 repairs the audit-loop correctness defects that can make DCC
claim a finding was handled when it was not, or associate an audit row with the
wrong finding. It also establishes reliable, warning-free format and chunked
input coverage. It is a patch release: valid existing calls continue to work;
the new identity and disposition fields are additive.

## Scope

This release implements these Phase 1 and Phase 2 items from
`ENGINEERING_PLAN.md`:

1. Add a deterministic `finding_id` to every `dcc_findings` row.
2. Require every audit row to name its exact source `finding_id`.
3. Reconcile by `finding_id`, never by the ambiguous
   `record_id + check_id` pair.
4. Validate the complete action configuration before changing data.
5. Represent an explicitly unhandled finding as `unhandled`, rather than
   silently applying the historical default `flag` action.
6. Repair CSV/TSV chunk type locking and separator defaults, and cover the
   documented chunk boundary and encoding cases without warnings.
7. Cover the complete supported read-format matrix, including multi-batch
   Parquet and Feather inputs and a fixed SAS fixture.

Numeric parsing, missing-value scoring semantics, YAML identifier migration,
new user-facing workflow APIs, performance target certification, and a
ten-million-row benchmark remain outside this patch and are scheduled for
later Phase 1 / Phase 2 / 1.1.0 work.

## Public contract

`dcc_findings` gains an additive `finding_id` column.  It is deterministic for
the input finding order and is constructed from a run identifier, check ID,
record ID, variable, and zero-based occurrence within the same
run/check/record/variable tuple.  The constructor accepts an optional
`run_id`; direct callers that omit it receive the stable `manual` run ID.
`dcc_detect()` supplies a run ID derived from its source data and ruleset
metadata.  Empty findings retain the new column with character type.

`dcc_audit_log()` gains an additive `finding_id` column.  One finding can
produce multiple audit rows; every such row repeats its source ID.  A `flag`
action produces one audit row and is a terminal `flagged` disposition.  An
`exclude` action produces a terminal `excluded` disposition; `set_na` and
`recode` produce `changed`.

`dcc_reconcile()` returns one row per finding with `finding_id`, `action`,
`status`, and `handled`.  `handled` is true only for terminal statuses
`changed`, `excluded`, or `flagged`.  Findings absent from the audit log have
status `unhandled`; audit rows whose ID is absent from findings make
reconciliation fail with `dcc_reconcile_error` rather than being summarized
as a permissive attribute.

## Execution and validation

`dcc_execute()` performs validation before copying or modifying the input:

* `id_var` values must be present, non-missing, and unique.
* Every finding record ID and cell variable must exist for the action that
  needs it.
* `actions` must be a named list and every configured rule ID must occur in
  the findings.  Unknown IDs are errors.
* Action names are limited to `exclude`, `set_na`, `flag`, and structured
  `recode`; group-level (record-less) findings can only be flagged.
* A recode map must be named and contain the observed old value for every
  finding to which it applies.  A missing map entry is an error, not a skipped
  update.

The historical `default = "flag"` remains accepted for compatibility but does
not auto-handle unmatched findings in 1.0.1.  Findings without an explicit
action are recorded as `unhandled`, no data is changed for them, and the
result can be inspected through `dcc_reconcile()`.  A future minor release may
introduce an explicit preview validator; this patch keeps the behavior local
to execution and reconciliation.

## Format and chunking contract

`dcc_detect_chunked()` infers a tab separator for a `.tsv` path when `sep` is
not supplied, and a comma separator for `.csv` and `.txt`; an explicit `sep`
always wins. The first CSV/TSV chunk establishes column types using the
data.table `colClasses` contract of a named list from type to column names.
All later chunks receive that list. This prevents the current “column name not
found” warnings while retaining the first-chunk type lock.

The chunked CSV/TSV suite covers quoted delimiters, embedded newlines, exact
chunk-size multiples, a later all-missing chunk, and UTF-8/latin1 inputs. Each
case must produce the same finding rows as in-memory detection and no warning.

Arrow validation runs both Parquet and Feather through multiple record batches
and compares each to in-memory detection. The fixture has integer, floating,
character, logical, date, and missing-value columns. The format matrix covers
CSV, TSV, JSON, Excel, SPSS, Stata, SAS, Parquet, and Feather through
`dcc_read()`. A fixed synthetic `.sas7bdat` fixture is stored under the test
fixtures directory; it is read, never generated during the suite. Excel, SPSS,
and Stata fixtures are generated by test helpers using declared test-only
dependencies; CI installs all declared Suggests dependencies.

The current developer environment installs Arrow and the optional readers so
the matrix runs with no skips. The package retains its optional-dependency
runtime guidance for ordinary users who do not install a particular reader.

## Data flow and error handling

`dcc_findings()` / detector output creates IDs -> `dcc_execute()` validates
the full plan -> each finding either writes audit rows carrying its ID or is
recorded unhandled -> `dcc_reconcile()` verifies a one-to-many exact ID join.
For file inputs, `dcc_read()` or `dcc_detect_chunked()` selects the format
reader, locks or inherits a stable schema, and feeds the same detector path.
No raw input is modified; failures occur before any result object is returned.
All new validation errors use the existing `dcc_execute_error` class, and
reconciliation integrity failures use `dcc_reconcile_error`.

## Compatibility and documentation

Existing finding and audit columns retain their names and meanings.  The new
columns are appended, so callers selecting known columns remain valid.  Tests,
Rd files, README release status, NEWS, and CHANGELOG will be updated to avoid
claiming unimplemented Phase 1 work.

## Testing

Tests are written before production changes and must demonstrate a genuine
failure first.  Coverage includes duplicate findings on a cell, multiple audit
rows for one finding, a missing audit row, a stray audit row, unknown action
ID, duplicate/missing data IDs, invalid record/variable/action combinations,
unmapped recodes, the full read-format matrix, and CSV/TSV plus Arrow chunked
equivalence. The relevant unit tests and the complete package suite must pass
with zero failures, warnings, and skips before release work proceeds.
