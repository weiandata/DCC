# DCC 1.1.0 Survey-Staff Usability Implementation Plan

> For agentic workers: REQUIRED SUB-SKILL: Use superpowers:executing-plans
> task-by-task. Steps use checkbox syntax for tracking.

**Goal:** Let survey staff run a full, safe cleaning workflow without writing
YAML or understanding data.table: a one-command `dcc_run()` with a fixed
output layout, an Excel cleaning-plan configuration, and structured validators
that explain what failed, why, and how to fix it. Preview is the default and
raw input is never overwritten.

**Architecture:** `R/run.R` owns `dcc_run()` orchestration and the fixed output
writer. `R/config.R` owns `dcc_read_config()` (Excel -> internal rules +
actions) and `dcc_validate_config()`. `R/validate.R` owns
`dcc_validate_rules()`, `dcc_validate_data()`, and `dcc_doctor()`. All reuse
the 1.0.1 engine (`dcc_read`, `dcc_detect`, `dcc_execute`, `dcc_report`,
`dcc_manifest`, `dcc_reconcile`); none re-implements detection or execution.

**Tech Stack:** R (>= 4.1), data.table, readxl/writexl (Excel config and
findings workbook), yaml, testthat edition 3, hand-maintained Rd and NAMESPACE.

## Global Constraints

- Work on a `codex/v1.1.0-survey-usability` branch; never stage `.codegraph/`.
- Additive only; reuse the existing engine and error classes.
- The raw input file is never modified or overwritten in any mode.
- Preview is the default mode; execution requires an explicit `mode`.
- Every validator returns a structured report *and* changes nothing.
- New exports get Rd + NAMESPACE entries; new column symbols go into
  `globalVariables()`.
- Full suite stays 0 failure / 0 warning / 0 skip with all Suggests installed.

### Task 1: Structured validators

**Files:** Create `R/validate.R`, `tests/testthat/test-validate.R`, Rd files;
modify `NAMESPACE`.

**Interfaces:**

- Produces: `dcc_validate_rules(rules)`,
  `dcc_validate_data(data, rules, id_var = NULL)`, and
  `dcc_doctor(data, rules, id_var = NULL)` -> a `dcc_validation` object: a
  data.frame with `code`, `severity` (`info`/`warn`/`fail`), `field`, `rows`
  (list column of affected row indices), and `fix` (a suggested correction),
  plus a `print` method that renders each issue in plain language.

- [ ] **Step 1 (red):** In `test-validate.R`, feed a rule set that references a
  missing variable and a data frame with a duplicated `id_var`; assert
  `dcc_validate_rules()` reports a `RULE_VAR_MISSING` fail with the offending
  field, and `dcc_validate_data()` reports `DATA_DUP_ID` with the affected
  rows. Assert a clean input yields an empty (all-`info`/none) report and that
  neither call changes its input. Confirm failure.
- [ ] **Step 2 (green):** Implement the validators over the existing rule and
  data structures, reusing `resolve_data()` and the rule field contracts from
  `eval_check()`. `dcc_doctor()` runs both and merges the reports.
- [ ] **Step 3:** Add `dcc_validation_errors(report)` (from the AI-usability
  plan) or a `fail`-only filter helper. Rd + NAMESPACE; commit
  `feat(run): add structured validators`.

### Task 2: Excel cleaning-plan configuration

**Files:** Create `R/config.R`, `tests/testthat/test-config.R`,
`inst/templates/cleaning-plan.xlsx` (generated by a helper script, checked in),
Rd files; modify `NAMESPACE`, `DESCRIPTION` (Suggests already include
readxl/writexl).

**Interfaces:**

- Produces: `dcc_read_config(path)` -> a `dcc_config`: a list with `rules`
  (a `dcc_ruleset`-compatible structure), `actions` (the action map), `id_var`,
  and `items`.
- Produces: `dcc_validate_config(config)` -> a `dcc_validation` report.

- [ ] **Step 1 (red):** In `test-config.R`, generate a small cleaning-plan
  workbook with `writexl` (sheets: `dataset`, `items`, `rules`, `actions`),
  read it with `dcc_read_config()`, and assert the resulting rules and actions
  equal a hand-written YAML equivalent (same `check_id`s, same action map).
  Confirm failure.
- [ ] **Step 2 (green):** Implement the workbook parser: map spreadsheet rows to
  `range`/`set`/`skip_logic`/detector rule structures and to the action map.
  Reject unknown columns and ambiguous rows with `dcc_config_error` rather than
  guessing.
- [ ] **Step 3:** Add `dcc_validate_config()` layering `dcc_validate_rules()`
  over the parsed rules. Ship a documented template workbook under
  `inst/templates/` and a helper that regenerates it.
- [ ] **Step 4:** Rd + NAMESPACE; commit
  `feat(run): read Excel cleaning-plan configuration`.

### Task 3: `dcc_run()` one-command workflow

**Files:** Create `R/run.R`, `tests/testthat/test-run.R`, `man/dcc_run.Rd`,
`man/dcc_run_files.Rd`; modify `NAMESPACE`.

**Interfaces:**

- Produces: `dcc_run(data, config, output_dir, mode = "preview")` -> a
  `dcc_run` object recording the mode, the resolved config, and the paths it
  wrote; `dcc_run_files(run)` returns those paths.
- Writes the fixed layout: `cleaned-data.csv`, `findings.xlsx`,
  `audit-log.csv`, `management-report.html`, `audit-report.html`,
  `manifest.yaml`, `run-summary.txt`.

- [ ] **Step 1 (red):** In `test-run.R`, run `mode = "preview"` on a small CSV
  plus a config and assert: the input file's bytes are unchanged, no
  `cleaned-data.csv` is written from raw input silently, `findings.xlsx` and
  `management-report.html` exist, and `dcc_run_files()` lists them. Then run
  `mode = "execute"` and assert `cleaned-data.csv`, `audit-log.csv`, and
  `manifest.yaml` appear and that `dcc_reconcile()` on the run is clean.
  Confirm failure.
- [ ] **Step 2 (green):** Implement orchestration: `dcc_read()` (or accept a
  `dcc_data`), `dcc_validate_*`, `dcc_detect()`, then in execute mode
  `dcc_execute()`; write reports with `dcc_report()`, the findings workbook
  with `writexl`, the audit log with `dcc_export_log(format = "csv")`, and the
  manifest with `dcc_manifest()`. Add a `run-summary.txt` with counts and
  hashes. `verify` reconciles an existing run; `rerun` calls `dcc_rerun()`.
- [ ] **Step 3:** Assert every mode leaves raw input untouched (a dedicated
  test hashes the input before and after each mode).
- [ ] **Step 4:** Rd + NAMESPACE; commit
  `feat(run): add dcc_run() one-command workflow`.

## Acceptance

- `dcc_run()` writes the exact fixed layout and never overwrites raw input in
  any mode (test-enforced).
- Excel config round-trips to the same rules and actions as the YAML
  equivalent.
- Validators return structured, actionable reports and change nothing.
- Preview is the default; execution is explicit.
- Package checks and tests are 0/0/0 with no skips.

Note: the formal usability acceptance in `ENGINEERING_PLAN.md` section 4 Phase 4
(eight non-programmer participants, SUS >= 75) is a human process run outside
this code plan; it gates the *claim* that non-programmers can use DCC, not the
merge of these functions.
