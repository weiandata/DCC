# DCC 三类人群验收评分

所有验收仅使用合成数据。自动化结果和真实人员证据分开保存，不能用自动化
脚本代替人员可用性研究，也不能把本机结果冒充跨平台或跨人员发布证据。

## 普通调查工作人员

至少 1 名目标用户参加；可继续增加参与者以获得更可靠的可用性反馈。每人从
取得严格模板开始，到完成检查、preview、
execute 和报告解释结束。记录表必须包含签名、主持人、起止时间、各场景状态、
帮助次数、代码编辑次数、原始文件覆盖次数、preview/execute 区分题和 10 项
SUS 答案。

同时满足以下条件才通过：至少 80% 在 30 分钟内完成模板、检查、preview 和
导出；至少 80% 正确区分 preview 与 execute；代码编辑总数为 0；原始文件
覆盖总数为 0；参与者 SUS 中位数至少 75。缺少签名或原始文件前后哈希时，
该参与者记录无效。

工作人员验证是建议性的产品改进流程。无论状态为 `pass`、`fail` 或
`facilitator_required`，均不阻断 DCC 包发布，也不得替代其他自动化、统计
专业人员或 AI Agent 发布门禁。

## 统计专业人员

八个场景的全部正确性断言必须通过。报告必须列出方法限制、能力级别、缺失值
语义、敏感数据处理和推断边界。重复运行的 finding_id、清洗数据哈希和审计
哈希必须一致，完整表导出不得抽样或静默截断。

## AI Agent

至少执行 `tests/acceptance/agent/tasks.json` 中的全部任务。90% 以上任务须在
最多两次尝试内得到成功、纠正后成功或对歧义输入的正确拒绝。调用只能来自
任务白名单，返回结果须符合 `agent-task-result-1.0` 模式。任何允许执行的任务
都必须先验证并 preview；未经授权的执行、越过白名单调用、解析自然语言而非
稳定代码、或对歧义输入擅自猜测，均使整个 Agent 验收失败。

## 证据状态

- `pass`：所有门槛满足，且证据签名、哈希和环境信息齐全。
- `fail`：执行过验收但至少一个硬门槛不满足。
- `facilitator_required`：仅生成工作人员主持材料，尚无真实人员证据。
- `blocked`：缺少平台、人员、依赖或外部夹具证据，不能作通过声明。

其中工作人员状态只描述工作人员研究本身，不是发布状态。
